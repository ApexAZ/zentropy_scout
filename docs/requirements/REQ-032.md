# REQ-032: File-Level Docstring Header Audit (Frontend & Backend)

**Status:** Not Started
**Version:** 1.1
**PRD Reference:** Internal -- Code Organization & Maintainability
**Backlog Item:** N/A
**Last Updated:** 2026-03-31

> **Note:** This REQ supersedes `REQ-032_frontend_jsdoc_import_hygiene.md` (v0.1),
> which covered only `lib/`, `hooks/`, and `types/` (~55 files) plus import direction
> fixes. Version 1.0 expanded scope to all frontend files (~262 in-scope). Version 1.1
> adds backend Python files (~194 in-scope). The import direction fixes from v0.1 are
> already complete and are not repeated here.

---

## 1. Purpose

Every non-test source file in `frontend/src/` and `backend/app/` must have a
three-axis file-level docstring header so an LLM reading any file cold can answer:
what it does, where it sits in the system, and what other files it coordinates
with -- without opening any other file.

---

## 2. Standard

### 2.1 Frontend (TypeScript)

See `zentropy-docs` skill SS TypeScript File-Level Headers for the required template,
three-axis pass/fail test, and skip conditions.

### 2.2 Backend (Python)

Google-style module docstrings with `Coordinates with:` and `Called by:` sections,
as established by REQ-031 for service files. Template:

```python
"""One-line summary of what the module does.

REQ-XXX SS Y.Z: Requirements traceability. Additional context about the
module's role and design decisions.

Coordinates with:
  - path/to/module.py: why (e.g., "imports FitScoreResult for batch aggregation")
  - path/to/other.py: why

Called by: path/to/caller.py (context), path/to/other_caller.py (context).
"""
```

Three-axis pass/fail test (Python equivalent):
- **What**: First line of docstring answers "what does this module do"
- **Where**: REQ references identify which specification section this implements
- **With**: `Coordinates with:` and `Called by:` name specific module paths and
  explain the relationship

---

## 3. Scope

### 3.1 Frontend

All non-test `.ts` and `.tsx` files in `frontend/src/` except:

- `*.test.ts` / `*.test.tsx` (test files)
- `components/ui/` shadcn primitives that are thin Radix wrappers with no custom
  logic (dialog.tsx, radio-group.tsx, textarea.tsx, skeleton.tsx, select.tsx,
  switch.tsx, checkbox.tsx, tooltip.tsx, card.tsx, tabs.tsx, input.tsx, label.tsx,
  button.tsx, table.tsx -- verify each before skipping)

### 3.2 Backend

All non-test `.py` files in `backend/app/` except:

- Test files (`tests/` directory)
- Empty `__init__.py` files (no logic, no re-exports)
- Alembic migration files (`backend/alembic/versions/*.py`, `backend/alembic/env.py`)

Backend files fall into two categories:
- **Services (~86 files)**: Already have `Coordinates with:` / `Called by:` from
  REQ-031. Need verification that headers are accurate and complete.
- **Non-service files (~108 files)**: Have Google-style docstrings with REQ refs
  but no `Coordinates with:` / `Called by:` sections. Need additions.

---

## 4. Per-File Process

For each file:

1. Read the file to understand its logic
2. Trace its imports to identify what it depends on
3. Trace its exports / usage via Grep to identify what depends on it
4. Write the header -- all three axes must pass

---

## 5. Definition of Done

### Frontend
- Every in-scope file begins with `/** @fileoverview ... */`
- `Layer:` and `Feature:` fields populated correctly
- `Coordinates with:` lists specific relative paths with relationship explanation
- `Called by / Used by:` lists specific consumers
- All headers pass the three-axis test from `zentropy-docs` skill
- No functional code changes -- headers only
- Full frontend lint and typecheck pass at phase gates

### Backend
- Every in-scope file has a module-level docstring with all three axes
- First line clearly describes what the module does
- REQ references link to relevant specification sections
- `Coordinates with:` lists specific module paths with relationship explanation
- `Called by:` lists specific callers with context
- No functional code changes -- docstrings only
- `ruff check .` and `pyright` pass at phase gates
